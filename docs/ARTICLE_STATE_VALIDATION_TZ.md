# ТЗ: проверка состояния артикула перед изменениями из Google-таблицы

## 1. Цель

Перед изменением цены, скидки или остатка определить состояние артикула в указанном кабинете Wildberries и сообщить пользователю о проблеме непосредственно в заполненных им полях Google-таблицы.

На первом этапе проверяются только следующие состояния:

- артикул является активным и найден в кабинете;
- артикул находится в корзине WB;
- артикул не найден ни среди активных карточек, ни в корзине;
- состояние артикула не удалось проверить из-за ошибки WB API.

Новые колонки в Google-таблицу не добавляются.

## 2. Объём первого этапа

В первый этап входят:

1. Проверка артикула в том кабинете, который указан в строке Google-таблицы.
2. Поиск сначала среди активных карточек, затем среди карточек в корзине.
3. Запрет отправки изменений для артикула, который находится в корзине, не найден либо не был надёжно проверен.
4. Запись русскоязычной причины в заполненные поля изменения цены, скидки и остатка.
5. Игнорирование входных данных для изменения габаритов.
6. Логирование технических подробностей проверки и ответов WB API.
7. Автоматические тесты новой логики.

В первый этап не входят:

- подробный перевод всех ошибок изменения цены, скидки и остатка;
- проверка фактического применения новых значений после запроса;
- добавление колонок статуса или комментариев к ячейкам;
- восстановление карточки из корзины;
- изменение габаритов карточки.

## 3. Поля Google-таблицы

Обрабатываются как команды пользователя:

| Операция | Заголовок поля | Текущая колонка |
|---|---|---|
| Новая цена | `Установить новую цену` | J |
| Новая скидка | `Установить новую скидку %` | L |
| Новый остаток | `Новый остаток` | AF |

Нельзя полагаться только на буквенные адреса колонок. При чтении данных следует использовать заголовки. При записи допустимо сохранить текущий механизм адресации, если структура таблицы гарантированно фиксирована.

## 4. Габариты

Поля:

- `Новая\nДлина (см)`;
- `Новая\nШирина (см)`;
- `Новая\nВысота (см)`

не участвуют в обработке команд из Google-таблицы.

Требуемое поведение:

- значения этих полей не добавляются в данные на изменение;
- запрос изменения габаритов в WB API не выполняется;
- поля не очищаются после выполнения job;
- в поля не записываются статусы или ошибки;
- наличие любых значений или текста в этих полях не влияет на цену, скидку, остаток и состояние артикула.

На уровне получения команд из Google-таблицы блок `dimensions` должен быть замокан пустым результатом либо полностью исключён из результата. Предпочтительный контракт: не создавать ключ `dimensions` вообще.

Получение текущих габаритов карточки для отображения или других независимых сценариев этим требованием не запрещается. Отключается только редактирование габаритов на основании колонок `Новая ...`.

## 5. Источники состояния артикула

Для проверки используются методы WB Content API:

1. Активные карточки: `POST /content/v2/get/cards/list`.
2. Корзина: `POST /content/v2/get/cards/trash`.

Проверка выполняется отдельно для каждого кабинета с токеном этого кабинета.

Запросы должны быть пакетными: нельзя выполнять отдельный полный обход карточек для каждой строки Google-таблицы. Необходимо собрать уникальные `nmID` по кабинету и классифицировать их одним проходом с учётом пагинации API.

## 6. Модель состояний

Для внутренней логики вводятся четыре явных состояния:

| Состояние | Условие |
|---|---|
| `ACTIVE` | Артикул найден методом списка активных карточек |
| `IN_TRASH` | Артикул не найден среди активных карточек, но найден в корзине |
| `NOT_FOUND` | Оба поиска успешно завершились, но артикул не найден |
| `CHECK_FAILED` | Хотя бы один необходимый поиск не удалось надёжно завершить |

Приоритет классификации:

1. Если артикул найден среди активных карточек, результат — `ACTIVE`; запрос корзины для него не обязателен.
2. Для оставшихся артикулов выполняется поиск в корзине.
3. Найденные в корзине получают `IN_TRASH`.
4. `NOT_FOUND` допустим только при успешном завершении обоих необходимых поисков.
5. Таймаут, сетевой сбой, некорректный ответ, исчерпание повторных попыток или HTTP-ошибка не должны превращаться в `NOT_FOUND`.

Состояние определяется в рамках конкретного кабинета. Наличие того же `nmID` в другом кабинете не делает его активным для текущей строки.

## 7. Поведение перед изменением

Проверка состояния выполняется до отправки запросов изменения цены, скидки или остатка.

| Состояние | Отправлять изменения в WB | Действие в Google-таблице |
|---|---:|---|
| `ACTIVE` | Да | Продолжить существующую обработку |
| `IN_TRASH` | Нет | Записать сообщение в заполненные командные поля |
| `NOT_FOUND` | Нет | Записать сообщение в заполненные командные поля |
| `CHECK_FAILED` | Нет | Записать сообщение о временной ошибке |

Проверка состояния является общей для всех операций по артикулу и выполняется один раз за запуск для комбинации `кабинет + nmID`.

## 8. Сообщения пользователю

Фиксированные сообщения первого этапа:

| Состояние | Текст |
|---|---|
| `IN_TRASH` | `НЕ ВЫПОЛНЕНО: артикул находится в корзине WB` |
| `NOT_FOUND` | `НЕ ВЫПОЛНЕНО: артикул не найден в кабинете WB` |
| `CHECK_FAILED` | `ВРЕМЕННАЯ ОШИБКА: не удалось проверить артикул в WB` |

Пользователю не показываются:

- английский `errorText` WB;
- тело ответа в формате JSON;
- traceback;
- токен или другие секретные данные;
- внутренние имена исключений.

Оригинальные ответы и технические причины сохраняются в логах с указанием кабинета, `nmID`, endpoint и HTTP-статуса.

## 9. Правила записи сообщения в строку

Сообщение записывается только в те командные поля, которые были заполнены пользователем до начала обработки:

- заполнена новая цена — сообщение записывается в поле новой цены;
- заполнена новая скидка — сообщение записывается в поле новой скидки;
- заполнен новый остаток — сообщение записывается в поле нового остатка;
- заполнено несколько перечисленных полей — одинаковое сообщение записывается в каждое из них;
- пустое командное поле остаётся пустым;
- поля новых габаритов не изменяются ни при каком состоянии артикула.

Пример исходной строки:

| Новая цена | Новая скидка | Новый остаток |
|---:|---:|---:|
| 1500 |  | 7 |

Если артикул находится в корзине, результат должен быть таким:

| Новая цена | Новая скидка | Новый остаток |
|---|---|---|
| `НЕ ВЫПОЛНЕНО: артикул находится в корзине WB` |  | `НЕ ВЫПОЛНЕНО: артикул находится в корзине WB` |

## 10. Повторная обработка сообщений

Текст, записанный job, не должен распознаваться следующим запуском как новая команда.

Для первого этапа достаточно существующей числовой валидации командных полей при условии, что:

- цена обрабатывается только как допустимое числовое значение;
- скидка обрабатывается только как допустимое числовое значение;
- остаток обрабатывается только как допустимое целое числовое значение;
- служебный текст не приводит к исключению и не отправляется в WB API.

Пользователь повторяет операцию вручную: удаляет сообщение и вводит новое числовое значение.

## 11. Ошибки WB при проверке состояния

Минимальные правила классификации:

| Ситуация | Результат |
|---|---|
| HTTP 200 и корректный полный обход | Использовать найденные `nmID` |
| HTTP 401/403 | `CHECK_FAILED` для затронутого кабинета |
| HTTP 429 после исчерпания повторов | `CHECK_FAILED` |
| HTTP 500–599 после исчерпания повторов | `CHECK_FAILED` |
| Таймаут или ошибка соединения после повторов | `CHECK_FAILED` |
| Некорректный JSON или отсутствуют обязательные поля | `CHECK_FAILED` |
| Частично пройдена пагинация, затем произошла ошибка | Нельзя классифицировать оставшиеся артикулы как `NOT_FOUND` |

Допускается сохранить `ACTIVE` для артикула, который был однозначно найден до последующей ошибки пагинации. Для всех не найденных к моменту ошибки результат должен быть `CHECK_FAILED`.

## 12. Предлагаемый контракт результата

Проверка может возвращать структуру следующего смысла:

```python
{
    "CabinetName": {
        123456789: "ACTIVE",
        223456789: "IN_TRASH",
        323456789: "NOT_FOUND",
        423456789: "CHECK_FAILED",
    }
}
```

Точное техническое представление (`Enum`, dataclass или словарь) выбирается при реализации. Предпочтительно использовать `Enum` и не передавать по коду произвольные строки.

## 13. Изменения в потоке обработки

Ожидаемая последовательность:

1. Прочитать строки Google-таблицы.
2. Извлечь числовые команды цены, скидки и остатка.
3. Не извлекать команды изменения габаритов.
4. Сгруппировать уникальные артикулы по кабинетам.
5. Получить состояния артикулов через активный список и корзину.
6. Для неактивных состояний подготовить обновления с русским сообщением.
7. Передать в существующую логику изменений только `ACTIVE`.
8. Записать сообщения в Google-таблицу независимо от того, удалось ли перечитать полные данные карточки.
9. Продолжить существующую обработку успешных операций для активных карточек.

Запись сообщений о состоянии не должна зависеть от результата последующей актуализации карточки, базы данных, тарифов, фотографий или комиссий.

## 14. Совместимость с текущей логикой

На первом этапе существующее поведение успешных операций для `ACTIVE` сохраняется, кроме отключения редактирования габаритов.

Необходимо избежать текущей хрупкой зависимости, при которой Google-таблица обновляется только при наличии полных перечитанных данных карточки. Для записи ошибки состояния достаточно минимальной информации:

```python
{"Артикул": nm_id}
```

и набора целевых полей с сообщениями.

## 15. Логирование

Для каждого кабинета нужен итоговый лог:

- количество проверенных уникальных артикулов;
- количество `ACTIVE`;
- количество `IN_TRASH`;
- количество `NOT_FOUND`;
- количество `CHECK_FAILED`;
- количество артикулов, допущенных к изменениям;
- количество строк, для которых подготовлены сообщения.

Для ошибок API логируются endpoint, кабинет, HTTP-статус и безопасное тело ответа. Токены не логируются.

## 16. Требования к тестам

Минимальный набор автоматических сценариев:

1. Артикул найден в активном списке — операция допускается, сообщение состояния не записывается.
2. Артикул отсутствует в активном списке и найден в корзине — операция не отправляется.
3. Артикул отсутствует в обоих успешно полученных списках — `NOT_FOUND`.
4. Активный список вернул ошибку — `CHECK_FAILED`, а не `NOT_FOUND`.
5. Корзина вернула ошибку для не найденного активного артикула — `CHECK_FAILED`.
6. Ошибка возникла на следующей странице — ранее найденные карточки сохраняют подтверждённое состояние, остальные не становятся `NOT_FOUND`.
7. Заполнена только цена — сообщение записывается только в цену.
8. Заполнены цена, скидка и остаток — сообщение записывается во все три поля.
9. Пустые командные поля не изменяются.
10. Текст сообщения не воспринимается следующим запуском как команда.
11. Заполненные новые габариты игнорируются и остаются без изменений.
12. При наличии новых габаритов запрос изменения карточки не вызывается.
13. Проверка выполняется в правильном кабинете.
14. Одинаковый `nmID` в разных кабинетах классифицируется независимо.
15. Повторяющиеся строки одного артикула не вызывают повторного обхода WB API.

## 17. Критерии приёмки первого этапа

Работа считается выполненной, если:

- каждый артикул с командой цены, скидки или остатка предварительно классифицирован;
- карточки в корзине не отправляются на изменение;
- карточки, не найденные после двух успешных поисков, не отправляются на изменение;
- при проблемах проверки не формируется ложное сообщение «артикул не найден»;
- сообщения записываются на русском языке только в заполненные поля цены, скидки и остатка;
- новые поля Google-таблицы не добавлены;
- новые габариты полностью игнорируются и остаются нетронутыми;
- текущие габариты при необходимости продолжают читаться для независимых сценариев;
- служебные сообщения не запускают повторную операцию;
- технические ответы WB доступны в логах;
- тесты из раздела 16 проходят.

## 18. Последующее развитие

После завершения первого этапа на эту модель можно последовательно добавить:

1. Ошибки бизнес-валидации, включая отрицательную чистую прибыль.
2. Отсутствующий `chrtId` и ошибки складов.
3. Проверку фактического применения цены и скидки.
4. Русский справочник ошибок изменения по кодам WB.
5. Разделение окончательных и временных ошибок операций.

Эти расширения не должны менять контракт состояний артикула, определённый настоящим ТЗ.
