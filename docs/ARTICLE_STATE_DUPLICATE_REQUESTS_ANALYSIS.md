# Анализ дублирующихся запросов после ARTICLE_STATE-валидации

Дата анализа: 2026-07-19.

## Контекст

Анализ относится к незакоммиченным изменениям, добавляющим проверку состояния
артикула перед применением изменений из Google-таблицы.

Цель документа — зафиксировать найденные повторные запросы к Wildberries,
Google Sheets и PostgreSQL. В рамках анализа оптимизации не реализовывались.

## Краткий вывод

Сама ARTICLE_STATE-классификация устроена в целом разумно:

- активные карточки проверяются пакетным обходом `cards/list` по кабинету;
- `cards/trash` проверяется только для артикулов, не найденных среди активных;
- отдельный запрос для каждого артикула не выполняется.

Основное дублирование возникает после валидации: данные успешно изменённых
артикулов повторно собираются отдельно для Google Sheets, а затем ещё раз для
записи в БД.

## 1. Повторное чтение карточек WB

Для успешно изменённого артикула список карточек WB потенциально обходится три
раза:

1. ARTICLE_STATE-валидация вызывает `get_article_states_async()` и обходит
   `content/v2/get/cards/list`, а для неразрешённых артикулов — `trash`.
2. После применения изменений `add_new_data_from_table(...,
   only_edits_data=True)` снова получает карточки для обновления Google Sheets.
3. После возврата результата `CardActualizationService.actualize_card_data_in_db()`
   ещё раз получает те же карточки для обновления PostgreSQL.

Вместе со вторым и третьим обходами повторяются запросы цен, тарифов и комиссий.

### Возможная оптимизация

После применения изменений один раз собрать нормализованный post-write snapshot
и использовать его одновременно для:

- обновления Google Sheets;
- обновления таблиц `article`, `card_data` и `unit_economics` в PostgreSQL.

Это позволит убрать третий обход карточек и повторные запросы цен, тарифов и
комиссий.

## 2. ARTICLE_STATE отбрасывает уже загруженные данные карточек

`_find_requested_nm_ids()` получает от WB полные объекты карточек, но сохраняет
только найденные `nmID`. Статичные данные карточек затем запрашиваются повторно.

Потенциально вместе с состоянием можно возвращать найденный объект карточки или
его нормализованное представление. Его можно повторно использовать для полей,
которые не менялись текущей операцией:

- `vendorCode`;
- предмет;
- размеры;
- баркоды;
- фотографии.

Результат ARTICLE_STATE нельзя безусловно использовать как окончательный
post-write snapshot для динамических полей: цена или остаток могли измениться
после проверки.

## 3. Лишние запросы в режиме `only_edits_data=True`

После редактирования вызывается универсальный `add_new_data_from_table()`. Даже
при `only_edits_data=True` он получает:

- карточки;
- цены и скидки;
- тарифы;
- список складов;
- остатки по складам;
- комиссии по предметам.

Этот полный набор выполняется независимо от типа изменения. Например, после
изменения только цены запрашиваются склады и остатки, а после изменения только
остатка — тарифы и комиссии.

### Возможная оптимизация

Собирать post-write данные по фактически успешным операциям:

- после цены или скидки перечитывать цены;
- после остатка перечитывать остатки только при необходимости;
- статичные данные брать из ARTICLE_STATE snapshot либо из одного content-запроса;
- не перечитывать тарифы и комиссии, если предмет и габариты не менялись.

Так как изменение габаритов сейчас отключено, получение тарифов после обычного
изменения цены или остатка выглядит явно лишним.

## 4. `add_data_in_db=False` не полностью отключает работу с БД

При post-write обновлении `add_new_data_from_table()` вызывается с
`add_data_in_db=False`. Сейчас этот флаг влияет только на формирование списка
новых артикулов.

Если `result_nm_ids_data` не пуст, функция всё равно выполняет:

- `ArticleTable.check_nm_ids()`;
- потенциальный `ArticleTable.update_articles()`.

После этого `CardActualizationService` снова обновляет `article` вместе с
`card_data` и `unit_economics`. В результате возможны лишняя проверка БД и
повторная запись в `article`.

### Возможная оптимизация

Весь блок записи из `add_new_data_from_table()` должен зависеть от
`add_data_in_db`, а не только формирование `nm_ids_for_add`.

Это наиболее локальная и очевидная будущая оптимизация.

## 5. Глобальные выборки PostgreSQL перед разбором правок

Перед чтением команд из Google Sheets загружаются:

- все артикулы через `ArticleTable.get_all_nm_ids()`;
- все `chrtId` через `CardData.get_chrt_ids()`.

При этом изменения могут быть заполнены только для нескольких строк.

### Возможная оптимизация

1. Прочитать строки Google Sheets.
2. Собрать заполненные пары `(account, nmID)`.
3. Запросить из БД только соответствующие артикулы и нужные `chrtId`.

Для этого потребуется изменить текущий контракт `get_edit_data()`, поскольку
сейчас он принимает уже полностью загруженные словари из БД.

## 6. Повторное полное чтение основной Google-таблицы

В одном цикле обработки правок основная таблица полностью читается как минимум
дважды:

1. `get_edit_data()` вызывает `get_all_values()` для получения команд.
2. `update_rows()` вызывает `get_all_records()` перед записью результатов и
   сообщений ARTICLE_STATE.

### Возможная оптимизация

При первом чтении сохранить:

- номер строки;
- `account`;
- `nmID`;
- индексы целевых колонок.

Затем можно формировать адресные batch-update без второго полного чтения.

Повторное чтение частично защищает от перестановки строк пользователем во время
WB-запросов. Более безопасный вариант оптимизации — сохранить ожидаемый номер
строки, а перед записью точечно проверить пару `ЛК + Артикул`.

## 7. Повторное чтение статусов Google Sheets

За полный запуск статусный лист проверяется несколько раз:

1. в `add_actually_data_to_table()`;
2. в `check_edits_columns()`;
3. в `change_cards_and_tables_data()`.

Последнее чтение явно дублирует данные, уже полученные в
`check_edits_columns()`.

### Возможная оптимизация

Прочитать статусы один раз в начале job и передавать snapshot статусов в
последующие функции.

## Рекомендуемый порядок будущих изменений

1. Сделать так, чтобы `add_data_in_db=False` действительно исключал проверку и
   запись в PostgreSQL.
2. Передавать уже прочитанные статусы Google Sheets в
   `change_cards_and_tables_data()`.
3. Разделить post-write refresh по типам успешно применённых операций.
4. Сформировать единый post-write snapshot для Google Sheets и PostgreSQL.
5. После этого оптимизировать полные чтения Google Sheets и глобальные выборки
   `get_all_nm_ids()` / `get_chrt_ids()`.

Наибольший ожидаемый эффект дадут устранение повторного post-write сбора данных,
отказ от ненужных WB-запросов по типу операции и исправление поведения
`add_data_in_db=False`.
